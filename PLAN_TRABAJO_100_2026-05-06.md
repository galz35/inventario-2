## 3. Hallazgos criticos

### 3.1 Produccion no refleja todo el codigo fuente actual

El codigo fuente actual tiene rutas de bodega:

- `GET /api/v1/bodega/pendientes`
- `POST /api/v1/bodega/despachar`

Pero el proceso PM2 publicado responde `404` para:

- `http://127.0.0.1:3023/api/v1/bodega/pendientes?pais=NI`

Los logs de PM2 del ultimo arranque no muestran mapeadas esas rutas. Esto indica que produccion esta corriendo una version anterior cargada antes del cambio o no se reinicio/publico despues de modificar el controlador.

Impacto: el modulo movil de despacho y la pantalla web de bodega pueden fallar aunque el codigo exista.

### 3.2 Build frontend falla

Comando probado:

```bash
npm run build
```

Directorio:

```bash
/opt/apps/inventario-portal/web
```

Error principal:

```text
Rolldown failed to resolve import "xlsx" from "web/src/pages/Dashboard.jsx"
```

`npm ls xlsx` dentro de `web` muestra `(empty)`, aunque `web/package.json` declara `xlsx`. Hay que reinstalar dependencias o corregir lock/node_modules.

Impacto: no se puede publicar una nueva version confiable del frontend hasta resolver esto.

### 3.3 Lint frontend falla

Comando probado:

```bash
npm run lint
```

Errores principales:

- Imports no usados en `Dashboard.jsx`, `LoginPage.jsx`, `SSOHandler.jsx`.
- `fetchInit` y `refresh` se usan antes de declararse en `Dashboard.jsx`.
- Warnings de dependencias faltantes en `useEffect`.
- `setError` sincronico dentro de `useEffect` en `SSOHandler.jsx`.
- `process` no definido en `vite.config.js`.

Impacto: hay riesgo de comportamiento inestable en React y el codigo no pasa validacion basica.

### 3.4 Tests backend fallan

Comando probado:

```bash
npm test -- --runInBand
```

Directorio:

```bash
/opt/apps/inventario-portal/api-nest
```

Resultado:

- `src/app.controller.spec.js` pasa.
- `src/auth/auth.service.spec.js` falla.

Causa:

`AuthService` ahora depende de `ConfigService`, pero el spec no lo provee en `Test.createTestingModule`.

Impacto: no existe una suite backend verde para validar cambios.

### 3.5 App movil esta incompleta

No se pudo ejecutar `flutter analyze` porque `flutter` no esta instalado en el servidor.

Archivos relevantes:

- `/opt/apps/inventario-portal/movil/lib/features/inventario/data/inventario_repository.dart`
- `/opt/apps/inventario-portal/movil/lib/features/inventario/presentation/pantalla_detalle_despacho.dart`

Problemas detectados:

- `PantallaDetalleDespacho` usa articulos simulados.
- Confirmar despacho solo hace `Future.delayed`, no llama al API real.
- `getPendientes()` llama `api/v1/bodega/pendientes` sin enviar `pais`.
- La API real espera `lineas`, pero algunos scripts antiguos usan `detalles`.
- Falta pantalla/flujo movil para detalle real de solicitud antes de despachar.

Impacto: movil no puede considerarse 100%.

### 3.6 Seguridad y autorizacion quedan incompletas

El backend usa cookies `user_carnet` y `user_pais` para sesion local. `AuthGuard` valida `UsuariosSeguridad`, pero hay endpoints sensibles sin rol explicito:

- Crear solicitudes.
- Aprobar/rechazar solicitudes, con validacion manual en service.
- Consultar inventario/kardex/solicitudes.

Riesgos:

- Falta una matriz formal de roles por endpoint.
- Falta logout del backend.
- CORS esta fijo a `https://inventario.rhclaroni.com` y `http://localhost:5173`, pero la publicacion real es `https://rhclaroni.com/portal/inventario/`.
- Cookies no tienen `secure` configurado explicitamente para produccion.

### 3.7 SSO tiene errores en logs

Logs recientes:

- `invalid signature` en SSO.
- `Error validating portal session in Inventario: connect ETIMEDOUT 190.56.16.85:443`

Impacto: el acceso via portal puede fallar intermitentemente o depender de secretos/URL no alineados con Portal.

### 3.8 Scripts de prueba estan desactualizados

Archivos:

- `/opt/apps/inventario-portal/test_backend.js`
- `/opt/apps/inventario-portal/test_solicitud_flow.js`

Problemas:

- Apuntan a `http://localhost:3000/api/v1`, pero produccion usa `3023`.
- Usan headers `X-Country-Code` y `X-User-Carnet`; el backend actual usa cookies.
- Esperan campos `id_articulo`, `id_almacen`, pero los SP actuales devuelven `IdArticulo`, `IdAlmacen`.
- Prueban endpoints de alertas no expuestos en el controller actual.

Impacto: no sirven como validacion real del sistema actual.

## 4. Plan de trabajo recomendado

### Fase 1: Estabilizar build y publicacion

Objetivo: poder compilar, reiniciar y publicar sin romper produccion.

Tareas:

- Reinstalar dependencias del frontend con `npm ci --legacy-peer-deps` en `/opt/apps/inventario-portal/web`.
- Verificar que `xlsx` quede instalado y resuelto por Vite.
- Corregir errores de lint que bloquean React:
  - Quitar imports/estados no usados.
  - Reordenar o envolver `fetchInit` y `refresh` con `useCallback`.
  - Ajustar dependencias de `useEffect`.
  - Corregir `vite.config.js` para ambiente ESLint.
- Ejecutar `npm run build` hasta quedar verde.
- Reiniciar `portal-inventario-api` desde PM2 para que cargue rutas actuales.
- Validar que `/api/v1/bodega/pendientes` deje de responder 404.
- Ejecutar `/root/iinv.bash update` solo despues de build verde.

Criterio de salida:

- `npm run build` verde.
- API PM2 reiniciada.
- `/portal/inventario/` responde 200.
- `/api-portal-inventario/api/v1/almacenes?pais=NI` responde 200.
- `/api-portal-inventario/api/v1/bodega/pendientes?pais=NI` responde 401/403 si no hay sesion, o 200 si hay cookie valida, pero no 404.

### Fase 2: Corregir pruebas backend y contrato API

  - SSO token valido/invalido.
  - Portal session valida/invalida.
  - Permiso de jefe/RRHH para aprobar/rechazar.

  - Puerto `3023`.
  - Cookies en vez de headers antiguos.
  - Campos `IdArticulo`, `IdAlmacen`.
  - Payload `lineas` para despacho.
- Agregar una prueba smoke de endpoints publicados por nginx.

Criterio de salida:

- `npm test -- --runInBand` verde.
- Script smoke documentado y ejecutable.
- Contrato JSON documentado por endpoint.

### Fase 3: Cerrar modulo web operativo

Objetivo: que la pantalla web cubra flujo completo sin acciones rotas.

Tareas:

Criterio de salida:

### Fase 5: Seguridad, roles y SSO

1. Resolver build/lint frontend.

3. Arreglar tests backend.
4. Validar flujo web completo.

8. Documentar operacion y rollback.

## 6. Definicion de 100%

El sistema puede considerarse 100% cuando se cumpla todo esto:

- Web compila sin errores.
- Backend tests pasan.
- App movil deja de usar datos simulados en despacho.
- SSO via Portal funciona sin errores de firma.
- Roles impiden acciones no autorizadas.
- Flujo completo funciona:
  - login
  - listar almacenes/articulos
  - consultar inventario
  - crear solicitud
  - aprobar/rechazar
  - listar pendientes de bodega
  - despachar
  - actualizar stock
  - consultar kardex
- Nginx publica web y API en rutas definitivas.
- `/root/iinv.bash update` funciona con la estructura real del repo.
- Existe smoke test posterior a despliegue.
- Existe procedimiento de rollback.
